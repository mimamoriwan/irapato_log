# イラパトログ — アイデア設計書（MVP確定版）

発想の切り口：**繰り返しのイライラだけをアイデアの種にする**
更新：2026年5月（壁打ち完了・MVP設計確定）

---

## 発端と課題意識

個人開発のアイデア探しをする中で、「たった1回感じたイライラ」に対してすぐにアプリ化・サービス化の検討を始めてしまう癖があることに気づいた。本来アプリ化する価値があるのは「何度も繰り返し感じているイライラ」であるはずだが、その区別が自分の中でできていなかった。

また、現在のワークフロー（Claudeプロジェクト）において、単発のイライラと繰り返しのイライラが整理されないまま深掘りが始まってしまい、アイデアの種としての価値を判断できていないという課題があった。

---

## このアプリが解決すること

> **「アプリを作る価値があるイライラかどうかを、回数で証明するアプリ」**

日常で感じた小さなイライラを気軽に記録し続けることで、「これは何度も体験しているイライラだ」という気づきを数値として可視化する。その気づきをもって初めて、アイデアの種として育てる判断ができる。

### ワークフローの変化

```
【従来（課題）】
1回イライラする
　↓ すぐ
「これアプリにできないか」→ Claudeに相談（単発でも深掘りしてしまう）

【イラパトログ導入後】
1回イライラする → 記録するだけ
2回目 → 記録するだけ
3回目 → 「同じやつだ」とAIが可視化してくれる
　↓ 初めて
「じゃあどんなアプリが考えられる？」→ Claudeに相談
```

---

## 設計方針

### ポジション

**「イライラの記録装置」＋「アイデア化のゲートキーパー」**

- 記録装置としての摩擦ゼロ設計
- 3回以上の繰り返しが確認された時点でアイデア検討を解禁するゲート機能

### 開発フェーズの考え方

- **まず**：自分自身が使って動作確認・手応えを検証する
- **手応えが確認できたら**：製品化に舵取り

Roomiesと同じ「検証ファースト」の哲学を適用する。

---

## MVPの機能設計（確定）

### 画面構成（2タブ）

| タブ | 役割 |
|---|---|
| 記録タブ | 一言メモを気軽に残すだけ。摩擦ゼロ |
| 分析タブ | 全ログをAIに渡して意味でグループ化。回数を表示 |

### 記録タブ（確定）

- **音声入力ボタン**：Web Speech API使用（追加コストゼロ・Safari対応）
- **テキスト入力**：一言で端的に記入（必須）
- **詳細入力**：状況や背景を追記（任意・折りたたみ式）
  - 詳細があるほどAIのグループ判定精度が上がる
- 記録時にAIは使わない。保存するだけ

### 分析タブ（確定）

- 手動で「分析する」ボタンを押したときのみAIが動く
- **全件送信方式**（MVP段階はシンプルに全件渡す）
- 全ログをAIに渡し、意味の近さでグループ化
- 表現が違っても同じ意味と判断されたものは同じグループに束ねる
- **3回以上＝アイデアの種として検討する価値あり**という判断基準

#### 出力イメージ

```
🌱 起動・レスポンスの遅さ　4件　★★★★☆
　→「アプリの起動が遅い」
　→「開こうとしたら時間がかかった」
　→「起動に手間取って忘れた」
　→「メモしようとしたら手間取った」

🌱 情報が見つからない　2件　★★☆☆☆
　→「検索しても欲しい情報が出てこない」
　→「また同じ情報を検索してしまった」
```

## 技術スタック（確定）

| 要素 | 選択 | 理由 |
|---|---|---|
| フロント | React + Vite | 既存スタックと合わせる |
| 音声入力 | Web Speech API | コストゼロ・追加実装不要 |
| データ保存 | localStorage → Supabase | 動作確認後に移行 |
| AI分析 | Gemini API（2.5 Flash） | コスト・速度のバランス |
| ホーム追加 | ブラウザ（MVP） → iOSネイティブ（製品化時） | Apple Developer登録済み |

### AIコスト試算（全件送信）

| 時期 | ログ件数 | 1回の分析コスト | 月10回分析時 |
|---|---|---|---|
| 1ヶ月後 | 〜150件 | 約$0.001〜0.006 | 数十円以内 |
| 3ヶ月後 | 〜450件 | 約$0.003〜0.07 | 数百円以内 |
| 6ヶ月後 | 〜900件 | 約$0.007〜0.14 | 数百円以内 |

→ **個人使用の範囲ではコスト問題なし。**

注意点：ログが増えるとレスポンスが遅くなるため、分析は非同期実行＋ローディング表示を実装する。

---

## 意図的に削ぎ落とした機能（MVPではやらないこと）

| 非搭載機能 | 理由 |
|---|---|
| 記録時のリアルタイムAI仕分け | コスト増・摩擦増。分析タブで十分 |
| 「同じ？違う？」の毎回確認フロー | 摩擦ゼロ設計と矛盾する |
| アプリアイデア自動生成機能 | 使い込んでから手応えで判断。今は不要 |
| 認証・ログイン | MVP段階は自分専用。不要 |
| グループ化のUI操作 | AIに任せる |
| 週次スケジュールの自動分析 | 手動トリガーで十分 |

## MVP後に判断する機能（保留）

### アプリアイデア生成オプション

3回以上のグループに対して「このイライラを解消するアプリを考える」ボタンを置く。

```
📋 Claudeに貼り付け用テキスト（案）

【繰り返されているイライラ】
起動・レスポンスの遅さ（4件）

【具体的な記録】
・アプリの起動が遅い
・開こうとしたら時間がかかった
・起動に手間取って忘れた
・メモしようとしたら手間取った

→ このイライラを解消するアプリのアイデアを考えてください
```

Claudeプロジェクトに貼るだけで壁打ちが始まる。追加のAI APIコストを発生させずに手運用で繋げる設計。

**判断タイミング**：MVP使用後、「次のアクションへの導線が欲しい」と自分が感じたら追加する。

### 音声入力の製品化対応

- MVP：Web Speech API（無料）
- 製品化時：iOS標準SFSpeechRecognizer（無料・精度高）
- Whisper APIは使わない限り課金機能にする必要なし

## このアプリ自体がイラパトログの設計思想の実証

このアプリのアイデア自体が「何度も感じてきた自分のイライラ」から生まれている。単発のアイデアではなく、繰り返しの課題に対する解決策であることが、開発着手の判断根拠になっている。

---

## 総合評価

| 項目 | 評価 | 根拠 |
|------|------|------|
| 自己課題との一致 | ◎ | 発端が実体験・繰り返しの課題 |
| 開発難度 | 低 | React + Gemini API・シンプルな2タブ構成 |
| AIコスト | ◎ | 全件送信でも個人使用範囲では問題なし |
| 製品化ポテンシャル | ○ | 個人開発者・PM層に刺さる可能性あり |
| 検証の容易さ | ◎ | 自分で使えばすぐ分かる |

→ **自分専用MVPを最速で作り、手応えを確認してから製品化を判断する。**
